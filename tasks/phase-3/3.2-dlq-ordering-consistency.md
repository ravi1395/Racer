# Task 3.2 — DLQ Ordering Consistency

**Phase:** v1.6.0  
**Risk:** Low-Medium  
**Effort:** ~1 day  
**Breaking:** **Yes — behavior change** (`peekAll()` order reverses)

## Problem

`peekAll()` returns newest-first (LRANGE on LPUSH list) but `dequeue()` pops oldest-first (RPOP). Developers inspecting the DLQ see entries in a different order than what they'll get when reprocessing them — this causes confusion during debugging.

## File to Change

### `service/DeadLetterQueueService.java`

Reverse the `peekAll()` result to match dequeue order (FIFO — oldest first):

```java
// CURRENT:
return redisTemplate.opsForList()
    .range(RedisChannels.DLQ_KEY, 0, -1)
    .flatMap(json -> deserializeDlm(json).onErrorResume(e -> Mono.empty()));

// NEW:
return redisTemplate.opsForList()
    .range(RedisChannels.DLQ_KEY, 0, -1)
    .collectList()
    .flatMapMany(list -> {
        Collections.reverse(list);  // LPUSH stores newest at head; reverse for FIFO order
        return Flux.fromIterable(list);
    })
    .flatMap(json -> deserializeDlm(json)
        .doOnError(e -> log.warn("[RACER-DLQ] Skipping corrupted DLQ entry: {}", e.getMessage()))
        .onErrorResume(e -> Mono.empty()));
```

Note: The `doOnError` log line also fixes the current silent swallowing of corrupted entries.

## Migration Impact

**Any code that calls `peekAll()` and expects newest-first will break.** Before releasing:

1. Search all consumer codebases for calls to `peekAll()` or `racerDlq.peekAll()`
2. Update release notes with a prominent warning
3. Add a log note at startup if DLQ is enabled:
   ```
   [RACER-DLQ] peekAll() now returns messages in FIFO order (oldest first) as of v1.6.0
   ```
