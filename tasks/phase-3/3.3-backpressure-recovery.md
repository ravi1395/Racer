# Task 3.3 — Back-Pressure Recovery

**Phase:** v1.6.0  
**Risk:** Low  
**Effort:** ~2 days  
**Breaking:** No

## Problem

When `RacerBackPressureMonitor` activates and overrides the poll interval, the override is sticky — it never recovers automatically even if the queue drains below the threshold. Once throttled, the listener stays throttled until a restart.

## File to Change

### `backpressure/RacerBackPressureMonitor.java`

Add graduated recovery: when the queue fill ratio drops below threshold, step the poll interval down gradually (50% of the gap per check cycle) instead of snapping straight back to the original interval.

```java
// In the method that handles the "queue fill below threshold" branch:

long currentOverride = streamRegistrar.getBackPressurePollIntervalMs();
long originalInterval = racerProperties.getConsumer().getPollIntervalMs();

if (currentOverride > originalInterval) {
    // Graduated step-down: halve the excess each cycle to avoid thrashing
    long newInterval = Math.max(originalInterval, currentOverride / 2);
    streamRegistrar.setBackPressurePollIntervalMs(newInterval);
    log.info("[RACER-BACKPRESSURE] RECOVERING — poll interval reduced from {}ms to {}ms (target: {}ms)",
            currentOverride, newInterval, originalInterval);
} else {
    streamRegistrar.setBackPressurePollIntervalMs(0);  // fully recovered; 0 = use default
    log.info("[RACER-BACKPRESSURE] FULLY RECOVERED — poll interval restored to default ({}ms)",
            originalInterval);
}
```

### Recovery progression example

Given `originalInterval = 100ms` and an override of `3200ms`:

| Cycle | Poll Interval |
|-------|--------------|
| 0 (back-pressure triggered) | 3200ms |
| 1 (queue below threshold)   | 1600ms |
| 2 | 800ms |
| 3 | 400ms |
| 4 | 200ms |
| 5 | 100ms → fully recovered |
